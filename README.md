# Master Router

Мастер-агент: по запросу пользователя определяет нужного ИИ-агента и
проксирует к нему запросы. Все агенты — локальные сервисы на разных портах
с единым контрактом API (см. [Контракт агента](#контракт-агента)).

Мастер **stateless**: не хранит чаты, сообщения и фидбэк — всё это живёт у
агентов и скоупится по `X-User-Id`. Задача мастера — роутинг и проксирование.

## Архитектура

```
                 ┌─────────────────────────────────────┐
   клиент  ─────▶│            Master Router            │
                 │                                     │
                 │  route()   rule → embedding → llm   │
                 │  dispatch  forward / stream_chat    │
                 └───────┬───────────────┬─────────────┘
                         │               │
              ┌──────────▼───┐   ┌───────▼──────┐   ...
              │  агент ЕПоЗ  │   │  агент CFD   │
              │  :8443       │   │  :xxxx       │
              └──────────────┘   └──────────────┘
```

Мастер — единственная точка входа. Агенты сидят на localhost и доверяют
заголовку `X-User-Id`, который проставляет мастер (на текущем этапе — берётся
из заголовка клиента).

## Роутинг

`MasterRouter.route()` определяет агента каскадом из трёх слоёв. Каждый
следующий слой запускается, только если предыдущий не дал ответа:

1. **rule_router** — поиск по ключевым словам (`закуп`, `тендер`, `223-фз`
   и т.д.). Быстрый и детерминированный, ловит явные случаи.
2. **embedding_router** — семантическая близость запроса к описаниям агентов
   через `intfloat/multilingual-e5-small` (косинусная мера, порог отсечения).
3. **llm_router** — классификация запроса моделью `gemma2:2b` через Ollama,
   ответ форсится в JSON.

Если ни один слой не дал валидного агента, запрос уходит на `FALLBACK_AGENT`
(`chat`). Блокирующие вызовы (`encode`, `client.chat`) вынесены в
`asyncio.to_thread`, чтобы не блокировать event loop.

## Эндпоинты мастера

Все проксирующие эндпоинты требуют заголовок `X-User-Id: <uuid>` и
пробрасывают его агенту без изменений. Статус-коды и тела ответов агента
проходят насквозь (включая `204 No Content` и `404` на чужой ресурс).

### Служебные

| Метод  | Путь      | Назначение                              |
|--------|-----------|-----------------------------------------|
| `GET`  | `/agents` | Список включённых агентов (`id`, `name`)|
| `POST` | `/route`  | Определить агента по тексту запроса      |

`POST /route` — тело `{"message": "..."}`, ответ `{"agent": "epoz"}`.

### Сессии (прокси на агента)

| Метод    | Путь                                              |
|----------|---------------------------------------------------|
| `POST`   | `/agents/{agent_id}/sessions`                     |
| `GET`    | `/agents/{agent_id}/sessions`                     |
| `GET`    | `/agents/{agent_id}/sessions/{session_id}/messages`|
| `PATCH`  | `/agents/{agent_id}/sessions/{session_id}`        |
| `DELETE` | `/agents/{agent_id}/sessions/{session_id}`        |

### Чат (SSE-стрим)

| Метод  | Путь                                            |
|--------|-------------------------------------------------|
| `POST` | `/agents/{agent_id}/sessions/{session_id}/chat` |

Тело `{"message": "..."}`, ответ — поток `text/event-stream` (формат событий
см. [Контракт агента](#контракт-агента)).

### Фидбэк (прокси на агента)

| Метод    | Путь                                            |
|----------|-------------------------------------------------|
| `POST`   | `/agents/{agent_id}/messages/{message_id}/feedback` |
| `GET`    | `/agents/{agent_id}/messages/{message_id}/feedback` |
| `DELETE` | `/agents/{agent_id}/messages/{message_id}/feedback` |

### Авто-роутинг

| Метод  | Путь    | Назначение                                  |
|--------|---------|---------------------------------------------|
| `POST` | `/chat` | route → создать сессию → стримить, за раз    |

Тело:

```json
{
  "message": "...",
  "agent_id": "epoz",   // опционально: если задан — роутинг пропускается
  "session_id": 1       // опционально: если null — создаётся новая сессия
}
```

В заголовках ответа возвращаются `X-Agent-Id` и `X-Session-Id`. Клиент
передаёт их обратно в следующем запросе (`agent_id`, `session_id` в теле),
чтобы продолжить разговор в той же сессии того же агента (**sticky-сессии**).

## Контракт агента

Мастер форвардит по контракту, а не по конкретному агенту, — поэтому каждый
агент обязан реализовывать одинаковую форму API. Добавление нового агента,
соблюдающего контракт, требует только записи в `registry.py`.

Обязательная часть контракта:

- скоупинг всех ресурсов по `X-User-Id`; обращение к чужому ресурсу → `404`;
- `POST /sessions` возвращает объект с полем `id` (на него завязан `/chat`);
- `POST /sessions/{id}/chat` отдаёт SSE в формате:

  ```
  data: {"chunks": [{"text": "...", "source": "doc1.pdf", "score": 0.87}]}
  data: {"token": "RAG"}
  data: {"token": " — это"}
  data: {"message_id": 2}
  data: [DONE]
  ```

  - `chunks` — первое событие, извлечённые фрагменты-контекст (только при
    RAG-запросе; при small-talk отсутствует);
  - `token` — токены ответа, включая блок источников в конце;
  - `message_id` — ID сохранённого сообщения (для фидбэка);
  - `[DONE]` — завершение потока.

Полная спецификация ручек агента (`/sessions`, `/messages`, `/feedback`) —
едина для всех агентов и описана в документации агента.

## Реестр агентов

`registry.py` — единственное место, где задаются агенты:

```python
AGENTS: dict[str, AgentInfo] = {
    "epoz": AgentInfo(
        id="epoz",
        name="ЕПоЗ",
        url="https://127.0.0.1:8443",
        description="Закупки, ЕПоЗ",
    ),
    # ...
}
```

`AgentInfo` содержит `id`, `name`, `url`, `description`, `enabled`.

- `description` используется embedding- и llm-роутерами для классификации —
  чем содержательнее (с примерами запросов), тем точнее роутинг.
- `enabled=False` исключает агента из `/agents` и из роутинга.
- Агенты с пустым `url` отдают `503` при попытке проксирования.

> На текущем этапе реально запущен только **ЕПоЗ**. Остальным агентам имеет
> смысл выставить `enabled=False`, пока их сервисы не подняты, — иначе роутер
> может выбрать агента без рабочего `url`.

## Обработка ошибок

Проксирование стрима устроено так, чтобы различать сбои до и во время потока:

- агент **недоступен** или ответил `4xx/5xx` **до начала** стрима → клиент
  получает чистый `502` с понятным телом (статус ещё можно задать, поток не
  начат). Для этого первый чанк вычитывается до создания `StreamingResponse`;
- соединение оборвалось **в середине** стрима → статус уже отдан (`200`),
  поэтому в поток дописывается SSE-событие `data: {"error": "..."}`. Фронту
  достаточно проверять каждое событие на наличие поля `error`.


## Запуск

```bash
pip install -r requirements.txt

# модель для llm-роутера
ollama pull gemma2:2b

# запуск
python main.py
# или
uvicorn main:app --host 127.0.0.1 --port 8000 --reload
```

Мастер поднимается на `http://127.0.0.1:8000`. `timeout_keep_alive=300`
и `timeout=300` у httpx-клиента — под долгие SSE-стримы.

## Примеры запросов

```bash
# определить агента
curl -X POST http://127.0.0.1:8000/route \
  -H "Content-Type: application/json" \
  -d '{"message": "как подать заявку на тендер по 223-фз"}'
# {"agent": "epoz"}

# авто-роутинг + стрим (новая сессия)
curl -N -X POST http://127.0.0.1:8000/chat \
  -H "Content-Type: application/json" \
  -H "X-User-Id: 11111111-1111-1111-1111-111111111111" \
  -d '{"message": "что такое ЕПоЗ"}'

# продолжить ту же сессию (agent_id и session_id из заголовков прошлого ответа)
curl -N -X POST http://127.0.0.1:8000/chat \
  -H "Content-Type: application/json" \
  -H "X-User-Id: 11111111-1111-1111-1111-111111111111" \
  -d '{"message": "а какие сроки", "agent_id": "epoz", "session_id": 1}'
```
